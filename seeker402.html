402
HTTP 402 · Payment Required · RFC 9110 §15.5.3

Reserved since 1997. No phone has ever answered it.

x402 is the rail AI agents use to buy data and services over HTTP, per request, in stablecoins. It works — and it is entirely server‑to‑server. Machines in data centres paying machines in data centres. The consumer side does not exist. No phone can pay a 402.

The Seeker is the only phone on earth with a hardware key store built for exactly this. Nobody had checked whether Seed Vault could sign an x402 payment. So I checked. It can. Here is the evidence.

Instruction 00  ·  The proof

I ran the experiment before I wrote this deck

14 July 2026, on a physical Seeker, MWA 2.2.8, against my own live x402 server (AgentFeed — 15 paid routes, real mainnet USDC settling through the Coinbase CDP facilitator). I asked Seed Vault to sign a v0 transaction whose fee payer is the facilitator, not the user — the exact shape an x402 payment requires, and the thing everyone assumes a hardware wallet will refuse.

Yes — and that is the trap

Seed Vault signed a partially‑signed v0 transaction it does not pay the fee for. It supports v0. It injects nothing. The signing step — the one everybody assumed would fail — works.

And the single mutation it makes is the entire engineering problem, stated precisely: the client must implement @solana/kit's TransactionModifyingSigner, not TransactionPartialSigner. A partial signer staples the signature onto the original message — and the wallet's priority‑fee bump silently invalidates it. The payment fails verification with no useful error. Every developer who attempts this will hit that wall, and most will conclude it cannot be done.

Packaging x402 for React Native

To be exact about what is and is not done: the protocol question is answered — the wallet signs, and I know the one interface detail that makes it verify. What is left is packaging: x402's client is built on @solana/kit, which assumes a browser crypto stack React Native does not have. That is engineering, not research, and it is milestone one.

I am not applying to find out whether this works. I already know it works, and I know why it is hard.

Instruction 01  ·  What I will build

Seeker402 — the phone half of the machine economy

  1. seeker402 — the SDK. MIT, npm, React Native.

    The Seed Vault ↔ x402 bridge: an MWA‑backed TransactionModifyingSigner for @solana/kit, plus the React Native crypto layer x402 needs (RN has no crypto.subtle; associated‑token‑account derivation requires SHA‑256). Any Seeker app gains one call — await payFetch(url) — and can buy anything on the x402 rail.

  2. Session budgets — the primitive that makes micropayments usable

    You cannot ask for a fingerprint on every $0.001 call. So: one Seed Vault signature performs an SPL Approve, delegating a capped amount to an ephemeral session key. Calls then stream silently until the budget is spent. Funds never leave the user's wallet. Capped, revocable with Revoke, non‑custodial. This is new, and it is what turns a demo into a product.

  3. SKR settlement — an open‑source x402 facilitator

    Nothing in x402 requires USDC; the asset field is just an SPL mint. A facilitator that verifies and settles in SKR makes SKR spendable on machine services, from the phone.

  4. Reference app: skrly — already live on the dApp Store

    Not a demo I promise to build. A shipped app that becomes the first application in the world to pay HTTP 402 from hardware‑backed keys, buying live market data per query.

Instruction 02  ·  How SKR is used

Build the thing that makes other people spend it

SKR becomes the settlement currency for machine payments on mobile — payments, one of the four uses Solana Mobile names directly. Endpoints priced in USD, settled in USDC or SKR, with a developer‑configurable discount for paying in SKR. Every app that adopts seeker402 becomes an SKR sink. The session‑budget primitive works identically on an SKR token account.

I am explicitly not asking for SKR to distribute across an existing user base.
